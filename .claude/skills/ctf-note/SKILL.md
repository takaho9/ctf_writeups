# ctf-note — writeupを一般化した知識ノートに起こす

解いた個別問題のwriteupを、次に同種の問題へ当たったとき役立つ**一般化された知識ノート**へ変換する。
攻撃側（仕組み・手順・ペイロード）と防御側（検出・対策）の両面を含め、セキュリティエンジニアのポートフォリオとして通用する密度でまとめる。

## 使うタイミング
- 問題を解き、writeup（`docs/<ctf>/<category>/<challenge>.md`）を書き終えたあと
- ユーザーが「一般化して」「周辺知識をまとめて」「ノートにして」等と言ったとき

## 手順

### 1. 対象writeupを特定する
- 引数で問題名やパスが渡されていればそれを使う。
- 指定がなければ、`docs/` 配下で**最後に更新された writeup**（`docs/notes/` と `index.md`/`tags.md` は除く）を候補として提示し、これでよいか一言確認する。
- 対象ファイルを読み、次を抽出する: 問題名 / CTF / カテゴリ / 使った脆弱性・技術 / フラグ取得までの核心手順。
  - **フラグそのものや問題固有のシークレットはノートに含めない**（一般化が目的）。

### 2. トピックを決める
- writeupの中心となる脆弱性・技術を1つのトピック名にする（例: `SSTI`, `SQL Injection`, `ROT13 / 古典暗号`, `File Upload`）。
- 既に同トピックのノートが `docs/notes/` にあれば、**新規作成せず既存ノートに追記・更新**する（重複を避ける）。まず `docs/notes/` を確認する。

### 3. 周辺知識を裏取りする（Web検索）
- WebSearch / WebFetch で信頼できる一次情報を確認し、参考リンクを集める。優先: PortSwigger Web Security Academy、OWASP、HackTricks、PayloadsAllTheThings、公式ドキュメント、CVE/ベンダー勧告。
- **リンクは実在を確認したものだけ載せる**。URLを推測で書かない。検索できない場合はその旨を伝え、リンクは最小限にする。

### 4. ノートを執筆する
日本語で、以下の構成で書く（該当しない節は省略可）。攻撃と防御の両面を必ず含める。

```markdown
---
tags:
  - Notes
  - <トピック>      # 例: SSTI
  - <カテゴリ>      # 例: Web
---

# <トピック> — 一般化ノート

> 元にした問題: [<問題名>](../<ctf>/<category>/<slug>.md)

## 概要
<!-- この脆弱性/技術が何か、1〜2段落で -->

## なぜ成立するか（根本原因）
<!-- どういう実装ミス・仕様で起きるのか -->

## 一般化した攻撃手順
<!-- 特定の問題に依存しない、再現可能なステップ -->

## ペイロード / チートシート
<!-- 判定用・悪用用の代表的な入力。コードブロックで -->

## バリエーション・フィルタ回避
<!-- WAF/サニタイズ回避、別コンテキストでの応用 -->

## 検出と防御
<!-- 防御側の視点: 安全な実装、緩和策、ログ/検知の観点 -->

## 関連問題・関連トピック
<!-- このリポジトリ内の関連writeupへの [[リンク]]、関連する攻撃 -->

## 参考
<!-- 手順3で実在を確認したリンクのみ -->
```

内容は writeup の丸写しではなく**抽象化**する。writeup が具体例、このノートが原理・型、という関係にする。

### 5. ファイルを作成し nav に追加する
- ファイル名: トピックのkebab-case（例 `docs/notes/ssti.md`）。英数字が無いトピックは適宜ローマ字/英語スラッグにする。
- `mkdocs.yml` の `nav` の `- Notes:` セクション（インデント2）の末尾に、インデント6でエントリを追加する:
  ```yaml
    - Notes:
        - notes/index.md
        - <トピック> — 一般化ノート: notes/ssti.md
  ```
  タイトルに `:` 等 YAML特殊文字が含まれる場合はダブルクオートで囲む。
- `docs/notes/index.md` の「一覧」にも1行追記する。

### 6. 相互リンクする
- 元のwriteupの `## 学んだこと` などに、生成したノートへのリンクを1行追加する（例: `- 一般化ノート: [SSTI 一般化ノート](../../notes/ssti.md)`）。相対パスに注意。

### 7. 検証する
- `python -m mkdocs build --strict` を実行し、警告・エラーが無いことを確認する（Material for MkDocs のバナーは無視してよい）。`site/` は確認後に削除する。
- 追加したファイル・nav・相互リンクを簡潔に報告する。まだコミットはしない。

## 注意
- 出力言語は日本語（既存writeupに合わせる）。
- 参考リンクは実在確認したものだけ。フラグや問題固有の値はノートに含めない。
- 既存の同トピックノートがあれば新規作成せず更新する。
